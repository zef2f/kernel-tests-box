# Методика выполнения тестирования при помощи тестового набора hwmon-i2c-stub-tests

## Назначение тестового набора

Тестовый набор `hwmon-i2c-stub-tests` предназначен для функционального тестирования драйверов подсистемы `hwmon`, работающих через шину I2C, без использования реального аппаратного устройства. Проверка выполняется при помощи модуля ядра `i2c-stub`, который эмулирует I2C-устройство и позволяет предварительно записывать значения виртуальных регистров.

Актуальный перечень тестируемых драйверов определяется содержимым файла `list.txt` тестового набора. Для добавления нового драйвера в набор требуется:
- добавить сценарий `scripts/<driver>.sh`;
- добавить имя драйвера в `list.txt`;
- включить в конфигурацию ядра опцию, соответствующую тестируемому драйверу.

## Сборка ядра Linux

Сборка ядра Linux для выполнения тестов из набора `hwmon-i2c-stub-tests` проводится в соответствии с общей методикой сборки тестируемых ядер Linux, применяемой в стенде.

К базовым наборам опций конфигурации ядер для версии Linux 6.12 добавляются следующие общие опции, необходимые для выполнения тестов набора `hwmon-i2c-stub-tests`:

```bash
CONFIG_I2C=y
CONFIG_I2C_CHARDEV=y
CONFIG_I2C_STUB=m
CONFIG_HWMON=y
```

Кроме того, для каждого драйвера, входящего в `list.txt`, в конфигурацию ядра должна быть добавлена соответствующая опция. Например:

```bash
CONFIG_SENSORS_LM90=m
CONFIG_SENSORS_ADM1031=m
```

Если в набор добавляются новые драйверы, список соответствующих `CONFIG_*` должен быть расширен.

Для запусков выполнения тестов со сбором покрытия к конфигурации ядра добавляются следующие опции:

```bash
CONFIG_GCOV_KERNEL=y
CONFIG_GCOV_PROFILE_ALL=y
CONFIG_DEBUG_FS=y
```

После сборки ядра с указанными опциями для последующей обработки данных о покрытии сохраняется каталог сборки с полным путем к нему, содержащий файлы `*.gcno`:

```bash
find "$KBUILD_DIR" -name \*.gcno -print0 | xargs -0 tar cvfz gcov_build.tar.gz
```

где `KBUILD_DIR` - полный путь к каталогу сборки ядра с опциями для сбора покрытия.

## Подготовка тестового набора

Тестовый набор `hwmon-i2c-stub-tests` не требует отдельного этапа компиляции: набор состоит из shell-скриптов, запускаемых непосредственно в гостевой системе.

Тесты клонируются из репозитория:

```bash
git clone https://github.com/zef2f/hwmon-i2c-stub-tests.git
cd hwmon-i2c-stub-tests
```

## Подготовка корневого раздела гостевой системы

Запуск тестов выполняется в виртуальной машине с корневым разделом, построенным при помощи инструмента `debootstrap` на основе Debian 11 (bullseye).

Для выполнения тестов в гостевой системе должны быть доступны:
- `kmod`;
- `i2c-tools`;
- стандартное shell-окружение Debian;

Минимальный набор дополнительных пакетов, устанавливаемых в корневой раздел гостевой системы для выполнения тестов, может быть задан следующим образом:

```bash
i2c-tools
kmod
```

## Запуск qemu

Запуск тестов выполняется в виртуальной машине KVM, запущенной при помощи эмулятора QEMU. Используется образ загрузочного диска с корневым разделом, построенным при помощи `debootstrap`.

Тестовый набор `hwmon-i2c-stub-tests` не требует специальных дополнительных параметров QEMU по сравнению с базовым запуском стенда.

Пример вызова QEMU:

```bash
qemu-system-x86_64 \
    -cpu host -enable-kvm -m 4096 -smp 2 -nographic \
    -kernel "$PATH_TO_bzImage" \
    -append "console=ttyS0 root=/dev/sda earlyprintk=serial net.ifnames=0 console_msg_format=syslog" \
    -drive file="$ROOTFS",format=raw \
    -net nic,model=virtio,macaddr=52:54:00:12:34:58 \
    -net user,hostfwd=tcp::22000-:22 \
    -monitor none
```

где `ROOTFS` - полный путь к raw-образу загрузочного диска с корневым разделом, `PATH_TO_bzImage` - полный путь к образу тестируемого ядра.

## Запуск тестов

Запуск тестов выполняется в виртуальной машине от имени `root`, так как тестовый набор использует:
- `modprobe`;
- запись в sysfs;
- утилиты `i2cset` и `i2cget`.

При запуске без аргументов выполняются все тесты, перечисленные в `list.txt`:

```bash
cd /root/hwmon-i2c-stub-tests
./run-tests.sh
```

Для запуска подмножества тестов указываются имена драйверов в командной строке:

```bash
cd /root/hwmon-i2c-stub-tests
./run-tests.sh <driver1>
```

Во время выполнения тестов для каждого драйвера выполняются следующие действия:
- загружается модуль `i2c-stub` с параметрами, задающими виртуальное устройство;
- при необходимости подготавливаются значения регистров виртуального устройства при помощи `i2cset`;
- тестируемый драйвер привязывается к созданному устройству через `/sys/class/i2c-dev/.../new_device`;
- выполняется проверка sysfs-атрибутов драйвера;
- проверяются значения атрибутов, права доступа, диапазоны записи, alarm-биты и реакция драйвера на изменение регистров виртуального устройства.

Успешное завершение `run-tests.sh` соответствует нулевому коду возврата. Ненулевой код возврата означает наличие хотя бы одного неуспешного теста.

## Сбор покрытия

Для выполнения запуска со сбором покрытия перед запуском тестов необходимо смонтировать `debugfs` и, при необходимости, сбросить gcov-счетчики:

```bash
mount -t debugfs none /sys/kernel/debug
echo 0 > /sys/kernel/debug/gcov/reset
dmesg -C
```

После этого тесты запускаются обычным образом:

```bash
cd /root/hwmon-i2c-stub-tests
./run-tests.sh
```

После выполнения тестов данные покрытия из `/sys/kernel/debug/gcov` копируются и архивируются:

```bash
cp -r /sys/kernel/debug/gcov/ .
tar -cvf ./gcov_data.tar ./gcov/
```

Построение отчетов о покрытии в формате HTML выполняется при помощи `lcov` на машине, где доступны дерево исходных текстов ядра и каталог сборки ядра с файлами `*.gcno`, расположенными по тем же абсолютным путям, что и при сборке:

```bash
tar xvf gcov_data.tar -C "$GCDA_PATH"
geninfo --gcov-tool="$GCOV" -f -o gcov.info "$GCDA_PATH"
genhtml -t "$REPORT_TITLE" -o ./gcov-html ./gcov.info
```

где `GCOV` - путь к утилите `gcov` версии, соответствующей версии GCC, использованной при сборке ядра, а `GCDA_PATH` - путь к каталогу, содержащему перенесенные из виртуальной машины данные покрытия.
